## Option A: Brand Distinction Phase

### Core Question

Can AIL move from "strong, believable websites" to "more distinctive authored and branded websites" without breaking the current supported surface?

### What This Phase Would Focus On

- stronger authored voice on personal sites
- stronger signature-brand posture on company/product sites
- more visible visual identity systems
- less convergence toward safe but generic structure-first styling

### What Success Would Look Like

Success would mean:

- personal sites feel more clearly authored and less like refined templates
- company/product sites feel more recognisably branded and less like strong but safe product-homepage systems
- the strongest baselines become easier to distinguish from each other at first glance
- the system still preserves the current level of structure and runtime stability

### Safe First Cut

The safest first cut is:

- continue on the company/product line first

Why:

- it is the broadest mainstream proof line
- it has the most upside for higher distinction without needing new route families
- stronger brand distinction here improves both demo quality and market clarity

### Main Risk

The main risk is:

- pushing individuality too hard and losing the stable mainstream quality that currently makes the company line strong

### Best Use Case

Choose this phase if the next goal is:

- better demos
- stronger market differentiation
- more obviously authored output quality

## Option B: Customization UX Phase

### Core Question

Can AIL make the new durable customization path feel substantially easier, safer, and more guided for real users?

### What This Phase Would Focus On

- smoother hook discovery and selection
- less friction in scaffolding real overrides
- stronger workspace/project-level customization guidance
- selectively richer runtime context where it is high-value and stable
- more operator-facing clarity around durable customization workflows

### What Success Would Look Like

Success would mean:

- new users can discover and scaffold overrides without reading generated code
- operators can understand current hook surfaces from CLI and docs alone
- the most valuable override tasks are easy to start and hard to misuse
- the product feels more like a guided customization system, not just a set of internal mechanisms

### Safe First Cut

The safest first cut is:

- continue the current hook workflow UX rather than redesigning the boundary itself

Examples:

- guided hook selection from CLI suggestions
- cleaner workspace-level filtering and summaries
- selective expansion of runtime context only where it remains template-safe

### Main Risk

The main risk is:

- spending a lot of effort on convenience layers before the next phase direction is fully chosen

### Best Use Case

Choose this phase if the next goal is:

- real-user editability
- lower-friction extension workflows
- stronger post-generation product value

## Option C: Broader Platform Phase

### Core Question

Should AIL begin opening a broader product phase beyond the current website-first promise?

### What This Phase Would Focus On

- re-evaluating `app_min`
- deciding whether bounded app-like flows should become formal product scope
- deciding whether new route families or interaction patterns should enter the supported promise

### What Success Would Look Like

Success would mean:

- the product scope changes, not just the quality of the current scope
- the system can support a broader set of interaction surfaces without weakening the supported website mainline

### Safe First Cut

The safest first cut is:

- not implementation
- first do a formal scope/RFC pass that defines exactly what broader platform support would and would not mean

### Main Risk

The main risk is:

- opening a broader capability phase too early and diluting a mainline that is finally coherent

### Best Use Case

Choose this phase only if the next goal is:

- changing product scope itself
- not just making the current supported surface stronger

### Highest Immediate Demo/Market Payoff

- Option A: Brand Distinction Phase

### Highest Immediate Product-Usability Payoff

- Option B: Customization UX Phase

### Highest Scope Expansion Potential

- Option C: Broader Platform Phase

- Option B: Customization UX Phase

### Highest Strategic Upside If It Works

- Option C: Broader Platform Phase

Reason:

- it changes product scope itself
- but it is also the highest-risk path

## Recommended Order

The safest and most coherent order is:

1. Option B: Customization UX Phase
2. Option A: Brand Distinction Phase
3. Option C: Broader Platform Phase

Why:

- the current website frontier is already strong enough to hold
- the current customization milestone is now strong enough to deepen
- improving customization ergonomics strengthens the product without destabilizing the supported scope
- stronger brand distinction is valuable, but it is not as foundational as a better user modification loop
- broader platform work should wait until the current frontier is intentionally held, not while it is still being normalized

## Recommended Next Phase

If only one next phase opens now, the recommended choice is:

- **Customization UX Phase**

Reason:

- it builds directly on a new stable milestone
- it improves real user leverage instead of only internal proof quality
- it is the lowest-risk way to increase product value from the current frontier
- it keeps the supported mainline coherent

## Success Criteria For The Recommended Phase

If the next phase is Customization UX, success should mean:

1. users can discover likely hook targets faster
2. users can scaffold real override files with less ambiguity
3. workspace/project-level customization summaries become easier to use
4. the most valuable overrides require less manual code reading
5. the workflow stays template-safe and does not regress current website quality

## One-Line Recommendation

As of 2026-04-03, the best next phase is to deepen customization UX: the website frontier is already strong enough to hold, while the managed / unmanaged + hook workflow has just become stable enough that improving its ergonomics now offers the highest leverage at the lowest risk.
